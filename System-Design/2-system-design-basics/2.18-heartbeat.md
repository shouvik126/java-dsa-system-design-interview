# Heartbeat

## Background

In a distributed computing environment, workload and data partitioning are spread across multiple independent server nodes. To route requests effectively and maintain cluster health, nodes must maintain awareness of which peer servers belong to the system and whether those servers are operational and healthy.

In decentralized architectures, when a request arrives at a node, that node requires accurate, real-time cluster membership state to determine which server is responsible for handling the request. 

Timely detection of node failures is critical. It allows the system to take immediate corrective actions—such as re-routing incoming traffic, initiating leader election, and reallocating data shards to healthy instances—preventing cascading cluster degradation.

---

## Solution

A **Heartbeat** mechanism requires each server node to periodically transmit a lightweight signal (a "heartbeat" ping) to a central monitoring node or directly to peer nodes within the cluster to demonstrate that it remains active and operational.

![Heartbeat Protocol](assets/2.18-heartbeat.png)

Heartbeating is one of the primary failure detection mechanisms in distributed architectures:

- **Centralized Topologies**: Every worker node periodically sends a heartbeat message to a master or coordinator node (e.g., Kubernetes Control Plane, HDFS NameNode).
- **Decentralized / Peer-to-Peer Topologies**: Nodes periodically choose a subset of peer servers (often using gossip protocols) and exchange heartbeats every few seconds.

If a server fails to send a heartbeat within a predefined, configurable timeout threshold, the monitoring nodes mark it as suspected or dead. Upon declaring a node unreachable, the cluster stops routing requests to it, removes it from active membership lists, and triggers failover and replacement protocols.
