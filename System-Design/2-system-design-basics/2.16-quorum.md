# Quorum

## Background

In distributed systems, data is replicated across multiple servers to achieve fault tolerance and high availability. However, maintaining multiple data copies introduces a fundamental challenge: **How do we ensure consistency across all replicas so that clients always observe the most up-to-date data state?**

If a client writes value $v_1$ to replica $R_1$, but another client immediately reads from replica $R_2$ or $R_3$ before the update has replicated, the second client receives stale data. Establishing a **Quorum** solves this synchronization problem.

---

## Solution: What Is a Quorum?

In a distributed environment, a **Quorum** is the minimum number of cluster nodes that must successfully process and acknowledge a distributed operation before the system declares the operation overall successful.

For example, if a database is replicated across 5 machines, a quorum defines the minimum number of machines that must execute the same transaction outcome (commit or abort) to finalize the operation.

![Quorum Diagram](assets/2.16-quorum-diagram.png)

### Determining Quorum Size

To ensure majority consensus, a quorum requires agreement from **more than half** of the total nodes in the cluster ($N$):

$$\text{Majority Quorum} = \left\lfloor \frac{N}{2} \right\rfloor + 1$$

- **5-Node Cluster ($N=5$)**: A majority quorum requires at least **3 nodes** online.
- **4-Node Cluster ($N=4$)**: A majority quorum requires at least **3 nodes** online.

> [!TIP]
> **Why Odd Node Counts Are Recommended**: A 5-node cluster can tolerate **2 node failures** while preserving a majority (3 operational nodes). In contrast, a 4-node cluster can afford only **1 node failure** because losing 2 nodes leaves only 2 active nodes, failing the 3-node majority requirement. Therefore, production clusters should always be deployed with an **odd number of total nodes**.

---

## Quorum Formula ($W + R > N$)

To guarantee strong data consistency across reads and writes, a distributed system enforces the following quorum rule:

$$W + R > N$$

Where:
- $N$ = Total number of nodes in the replication group
- $W$ = Minimum number of write acknowledgments required (**Write Quorum**)
- $R$ = Minimum number of read responses required (**Read Quorum**)

When $W + R > N$, the set of nodes written to and the set of nodes read from are guaranteed to overlap by at least one node. That overlapping node will contain the latest written value.

```
       Write Nodes (W)                     Read Nodes (R)
[ Node 1 ]  [ Node 2 ]               [ Node 2 ]  [ Node 3 ]
                 \                       /
                  --->  Overlap Node  <---
```

### Common Quorum Configurations

| Configuration | Write Quorum ($W$) | Read Quorum ($R$) | Characteristics |
| :--- | :--- | :--- | :--- |
| **$N=3, W=2, R=2$** | $W=2$ | $R=2$ | **Strong Consistency** (Standard balanced setup) |
| **$N=3, W=1, R=3$** | $W=1$ | $R=3$ | **Fast Writes, Slow Reads** (Lower write durability) |
| **$N=3, W=3, R=1$** | $W=3$ | $R=1$ | **Slow Writes, Fast Reads** (High durability, read-optimized) |

### Key Guidelines for Quorum Design

1. **Full Replication ($R=1, W=N$)**: Requires all nodes to acknowledge a write. This approach is undesirable in dynamic environments because a single offline server causes all write operations to fail.
2. **Read-Heavy Optimization ($R \le W$)**: In most web applications, read volume far exceeds write volume. Setting $R \le W$ maximizes read performance and system availability.

---

## How It Works

1. **Majority-Based Quorum**: Operations require agreement from more than half of the operational cluster nodes. In a 5-node cluster, at least 3 nodes must participate to form a valid consensus.
2. **Read & Write Quorums**: Quorum thresholds can be tuned separately for read and write paths depending on application SLA requirements.

---

## Use Cases

- **Distributed Databases**: Engines like **Apache Cassandra**, **Amazon DynamoDB**, and **CockroachDB** use read/write quorums to balance latency against consistency.
- **Cluster Management**: Cluster orchestrators use quorums to elect active leaders and prevent **split-brain** scenarios, where a network partition isolates nodes into competing master sub-clusters.
- **Consensus Protocols**: Algorithms such as **Paxos** and **Raft** rely on quorums to achieve consensus regarding log replication and cluster state transitions.

---

## Advantages & Challenges

### Advantages
- **Fault Tolerance**: Sustains cluster operations cleanly despite node outages.
- **Data Consistency**: Prevents stale reads by ensuring write-read set overlap.
- **Tunable Availability**: Allows applications to balance availability and latency by adjusting $W$ and $R$.

### Challenges
- **Network Partitions**: During network degradation, forming a quorum may fail, impacting write or read availability.
- **Performance Overhead**: Waiting for multiple network acknowledgments introduces latency compared to single-node writes.
- **Operational Complexity**: Designing and managing quorums in dynamically scaling clusters requires careful configuration.

---

## Conclusion

Quorum is a fundamental concept in distributed systems architecture. By enforcing the $W + R > N$ rule, systems achieve strong consistency, fault tolerance, and high availability while balancing performance trade-offs.
