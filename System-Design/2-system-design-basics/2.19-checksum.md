# Checksum

## Background

In distributed systems, data is constantly transferred across network links and written to physical storage media. During data movement or storage, silent data corruption can occur due to storage hardware degradation, network packet corruption, memory errors, or software bugs. 

How can a distributed system guarantee data integrity so clients receive an error or fetch a clean copy instead of silently processing corrupted data?

---

## Solution

To ensure data integrity, the system calculates a **checksum** for a dataset and stores or transmits the checksum alongside the data itself.

![Checksum Verification Flow](assets/2.19-checksum.png)

### How Checksums Work

1. **Checksum Calculation**: A cryptographic or non-cryptographic hash function (such as MD5, SHA-1, SHA-256, SHA-512, or CRC32) processes the input data bytes and produces a fixed-length alphanumeric hash string—the **checksum**.
2. **Storage / Transmission**: When a system writes or sends data, it computes the checksum and attaches it to the data payload or stores it in metadata.
3. **Verification**: When a client or peer node retrieves data from storage or over the network, it recomputes the checksum of the received data and compares it with the stored checksum:
   - **Match**: The data is verified as untampered and uncorrupted.
   - **Mismatch**: Data corruption is detected. The client can reject the payload and automatically fetch a clean copy from an alternate replica.
